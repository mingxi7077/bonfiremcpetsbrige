# BonfireMCpetsBridge 功能需求说明

位置: bonfiremcpetsbrige 项目内说明书目录  
适用范围: MCPets 玩家宠物数据备份、快照、人工回滚与审计场景

## 1. 一句话理解这个插件

这个插件不改 MCPets 的核心玩法。
它做的事情是: 定时把 MCPets 当前主数据源中的玩家宠物数据拉出来, 备份到本地独立 SQLite 库, 并提供可审计、可校验、可人工回滚的恢复能力。

## 2. 项目背景

当前线上已经出现过玩家反馈: 宠物背包之前放入过物品, 重新上线后打开背包发现空白。

从现网排查结论看:

1. MCPets 当前主要运行在 MySQL 模式
2. `plugins/MCPets/PlayerData/*.yml` 不是当前主写入源
3. 需要额外的独立备份层, 用来做历史快照和人工恢复

因此本插件的目标不是替代 MCPets 存档, 而是在 MCPets 外层增加一个独立的保险层。

## 3. 核心目标

### 3.1 必须实现

1. 使用本地 SQLite 作为独立备份库
2. 每 1 小时自动备份一次 MCPets 玩家数据
3. 快照保留 7 天, 超期自动清理
4. 支持按玩家查看可用快照
5. 支持按玩家指定快照做手动回滚
6. 支持查看回滚任务状态
7. 回滚前自动生成一份“回滚前快照”
8. 回滚后做一致性校验并写入审计记录

### 3.2 明确不做

1. 不接管 MCPets 自身存档逻辑
2. 不自动替玩家做回滚
3. 不改 MCPets 的宠物 GUI 行为
4. 不修改玩家背包、末影箱等非 MCPets 数据
5. 第一阶段不支持把 MCPets Flat YAML 模式作为回滚写回目标

## 4. 数据范围

本插件以“玩家级 MCPets 数据快照”为基本单位。

每份快照至少覆盖以下字段:

1. 玩家 UUID
2. 快照时间
3. MCPets 玩家名称映射数据 `names`
4. MCPets 宠物背包数据 `inventories`
5. MCPets 宠物统计数据 `data`
6. 数据摘要哈希值
7. 快照来源说明

备注:

- 当前以 MCPets 主表中的 `uuid / names / inventories / data` 为标准恢复对象
- 这样可以保证恢复粒度与 MCPets 当前真实持久化结构一致

### 4.1 NBT 与高复杂物品兼容要求

生产环境内, 玩家放入 MCPets 背包的物品可能包含大量自定义 NBT 与跨插件元数据。

第一阶段必须把以下生态视为重点兼容对象:

1. `NBTAPI` 相关物品生态
2. `ItemsAdder` 自定义物品
3. `MMOCore` / MMO 系列衍生物品
4. `MythicMobs` 与 `ModelEngine` 生态掉落物
5. `CMI` 命令下发或写入 NBT 的物品
6. 其他依赖原始 NBT 标签识别行为的自定义物品

对应要求如下:

1. 桥接插件必须以“数据保真”为第一原则
2. 对 `inventories` 字段中的原始载荷只允许做读取、存储、哈希和原样写回
3. 第一阶段不得为了备份或回滚而主动把 `inventories` 反序列化为 Bukkit `ItemStack` 后再重新序列化
4. 第一阶段不得清洗、重排、格式化、裁剪或二次编码自定义 NBT 内容
5. 对 `names / inventories / data` 的快照必须按 MCPets 主数据源原始值落库
6. 回滚时必须按快照原始值原样写回 MCPets 主数据源
7. 插件必须兼容生产环境内存在 `NBTAPI` 的情况, 但第一阶段不要求把 `NBTAPI` 作为硬依赖

兼容边界也要写清楚:

1. 本桥接插件能保证“不额外制造二次数据损坏”
2. 如果某个物品在进入 MCPets 主数据源之前就已经被上游插件或 MCPets 本体错误处理, 桥接插件无法凭空修复那次上游损坏
3. 本桥接插件能恢复的是“已经成功进入 MCPets 主数据源并被快照采集到”的那份状态

## 5. 存储要求

### 5.1 备份库位置

默认放在:

`plugins/BonfireMCpetsBridge/backup.db`

### 5.2 存储隔离原则

1. 备份库必须与 MCPets 主数据库解耦
2. 备份库损坏不能反向影响 MCPets 正常运行
3. 备份库只作为桥接插件自己的快照仓库和审计仓库

### 5.3 保留策略

1. 快照保留 7 天
2. 清理任务至少每天执行 1 次
3. 对超过保留期的普通快照做自动删除
4. 回滚任务审计记录默认不跟随普通快照一起立刻删除, 便于排障

## 6. 备份策略要求

### 6.1 定时频率

1. 平均每 1 小时执行一次定时备份
2. 首次版本允许固定频率执行, 不要求复杂调度

### 6.2 备份方式

第一阶段优先采用“按玩家读取 MCPets 主数据源”的方式进行快照。

建议策略:

1. 周期任务扫描目标玩家集合
2. 从 MCPets 主数据源读取该玩家当前完整记录
3. 计算本次数据哈希
4. 若与最近一份快照完全一致, 可跳过落库
5. 若不同, 写入新快照

### 6.3 目标玩家范围

第一阶段建议以“最近上线过且存在 MCPets 数据的玩家”为主。

后续可扩展:

1. 只备份在线玩家
2. 备份全量 MCPets 玩家
3. 按白名单或黑名单过滤

## 7. 手动回滚要求

### 7.1 回滚前置条件

1. 必须由管理员手动执行
2. 默认要求目标玩家离线后执行
3. 必须指定明确的快照 ID
4. 必须在执行前验证目标快照是否存在
5. 必须在执行前生成一份“回滚前快照”

### 7.2 回滚对象

手动回滚恢复的是 MCPets 当前主数据源里的该玩家记录, 即:

1. `names`
2. `inventories`
3. `data`

### 7.3 回滚状态机

每次回滚任务必须可追踪状态, 最少包括:

1. `PENDING`
2. `VALIDATED`
3. `APPLYING`
4. `APPLIED`
5. `VERIFIED`
6. `FAILED`

### 7.4 回滚后的校验要求

至少校验以下内容:

1. 目标玩家主数据是否成功写回
2. 写回后读取的数据哈希是否与目标快照一致
3. 审计日志是否成功写入

## 8. 回滚影响逻辑说明

### 8.1 对玩家的影响

1. 只影响目标玩家的 MCPets 数据
2. 快照之后该玩家产生的新宠物数据变更会被覆盖
3. 不影响其他玩家数据
4. 不影响服务器主世界方块、经济、背包等其他插件数据

### 8.2 对 MCPets 主数据源的影响

会影响。

原因很简单:

- 回滚的本质就是把某个历史快照重新写回 MCPets 当前主数据源
- 如果当前 MCPets 使用 MySQL, 那么被修改的是 MCPets 的 MySQL 玩家数据记录

### 8.3 对本地 SQLite 备份库的影响

会影响, 但影响是可控且必要的。

具体包括:

1. 写入一条回滚任务记录
2. 写入一条审计日志
3. 写入一份“回滚前快照”
4. 可能写入一份“回滚后验证快照”或校验记录

也就是说:

- 本地 SQLite 会记录回滚发生过
- 但它不会因为执行回滚而丢失全部历史备份

### 8.4 是否会影响 MCPets 本地 YAML 数据

在当前线上 MySQL 模式下, 默认不会。

原因:

1. 当前 MCPets 主写入源是 MySQL
2. `plugins/MCPets/PlayerData/*.yml` 不是现网主存储源
3. 第一阶段桥接插件只对当前主数据源做回滚写回

如果未来某个环境切换成 Flat YAML 模式:

1. 第一阶段先不支持自动写回 YAML
2. 插件应识别到该模式后拒绝执行回滚
3. 在日志和命令返回里明确提示“不支持的主数据源类型”

## 9. 命令需求

第一阶段建议至少提供以下命令:

1. `/bmcb status`
2. `/bmcb backup now [player]`
3. `/bmcb snapshot list <player>`
4. `/bmcb snapshot info <snapshotId>`
5. `/bmcb rollback <player> <snapshotId>`
6. `/bmcb rollback status <jobId>`
7. `/bmcb cleanup run`

## 10. 日志与审计要求

必须记录:

1. 备份开始时间与结束时间
2. 备份扫描玩家数
3. 新增快照数
4. 跳过数
5. 清理删除数
6. 回滚操作者
7. 回滚目标玩家
8. 回滚前快照 ID
9. 回滚目标快照 ID
10. 回滚结果与失败原因

## 11. 安全要求

1. 默认只允许有管理权限的玩家或控制台执行回滚
2. 不允许模糊回滚, 必须指定快照 ID
3. 回滚前必须再次确认目标玩家身份与 UUID
4. 回滚流程中若校验失败, 必须标记为 `FAILED`

## 12. 第一阶段验收标准

满足以下条件即可进入开发阶段:

1. 能正常创建本地 SQLite 备份库
2. 能按 1 小时周期生成 MCPets 玩家快照
3. 能清理超过 7 天的快照
4. 能查询玩家的历史快照
5. 能执行一次成功的人工回滚
6. 能查询回滚状态并看到审计记录
7. 在现网 MySQL 模式下, 不会误改 `plugins/MCPets/PlayerData/*.yml`
8. 对带有 `NBTAPI / ItemsAdder / MMOCore / MythicMobs / ModelEngine / CMI NBT` 的物品, 桥接层不发生额外的 NBT 丢失或二次重编码损坏
